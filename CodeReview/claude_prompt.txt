### 特别关注的风险类型
==================================================

#### 崩溃风险
- 空指针 / nil 引用
- 数组 / 集合越界访问
- 类型转换失败
- 除零错误
- 未初始化的变量使用

#### 内存泄漏
- 循环引用
- 资源未释放（文件句柄、网络连接、定时器等）
- 监听器 / 观察者未移除
- 回调函数持有强引用

#### 线程安全
- 多线程访问共享可变数据
- 竞态条件
- 死锁风险
- 非主线程更新 UI（移动端）

#### 性能问题
- 循环中执行耗时操作
- 不必要的对象频繁创建
- 未优化的数据库查询
- 内存占用过大

#### 异常处理
- 错误处理缺失
- 异常被静默吞掉
- 错误信息不明确
- 异常传播路径不完整

#### API 使用
- 使用弃用 API
- 未做版本兼容性判断
- 使用私有或不稳定 API
- API 调用参数错误

#### 数据一致性
- 数据同步问题
- 事务处理不当
- 缓存与数据库不一致
- 并发修改导致数据损坏

==================================================
## 三、架构与设计层面风险（即使代码可运行也需要指出）
==================================================
- 职责是否混乱
- 是否存在业务逻辑直接写在 View / UI 层中
- 是否引入难以维护、测试或扩展的设计
- 命名是否存在歧义，影响长期维护
- 代码耦合度是否过高

==================================================
## 四、风险等级判定标准（用于统一严重程度）
==================================================
- 高  
  - 明确可能导致 crash、数据损坏、严重 UI 错乱
  - 明确的内存泄漏、死锁、线程安全问题
- 中  
  - 潜在性能问题
  - 并发使用不当但不一定立即触发
  - 可能导致用户体验下降
- 低  
  - 可维护性、可读性问题
  - 潜在但不紧急的隐患
  - 代码风格问题

==================================================
